Dynamic CDI Test Bean Addon
Auto-Mock Unsatisfied CDI Injection Points

A CDI portable extension that fills missing beans with Mockito mocks at bootstrap. One test dependency. Zero configuration.

CDI 4.1 Mockito 5 JUnit 6 Java 25+ Apache License 2.0

The Problem

If you've worked on a multi-module CDI project, you've seen this:

WELD-001408: Unsatisfied dependencies for type BaseDao<Order>
  at injection point [BackedAnnotatedField] @Inject private BaseService.dao

The container refuses to boot because a bean in your common module injects a type whose implementation lives in app. Production assembles everything fine — but your isolated module test is dead on arrival.

Consider a typical multi-module layout:

common/
  src/main/java/
    BaseEntity.java
    BaseDao.java              // abstract generic DAO
    BaseService.java          // @Inject BaseDao<E>
    AuditService.java         // interface — no impl here

app/
  src/main/java/
    Order.java                // extends BaseEntity
    OrderDao.java             // extends BaseDao<Order>
    OrderService.java         // extends BaseService<Order>
    AuditServiceImpl.java     // implements AuditService

BaseService<E> declares @Inject BaseDao<E>. When OrderService extends BaseService<Order>, the injection point resolves to BaseDao<Order> — satisfied by OrderDao in production. But when you test inside common, neither OrderDao nor AuditServiceImpl are on the classpath.

The usual workarounds are painful:

  • Manual mock producers — a @Produces method per missing type, maintained by hand, breaks when dependencies change.
  • Integration test modules — heavyweight, slow feedback.
  • Skip CDI tests entirely — defeats the purpose.

The Solution

Add Dynamic CDI Test Bean Addon as a test dependency. Every unsatisfied injection point is automatically filled with a Mockito mock. No configuration. No boilerplate. The container starts, your tests run. 

<dependency>
    <groupId>org.os890.cdi.addon</groupId>
    <artifactId>dynamic-cdi-test-bean-addon</artifactId>
    <version>1.0.0-SNAPSHOT</version>
    <scope>test</scope>
</dependency>

The extension registers itself via META-INF/services/jakarta.enterprise.inject.spi.Extension — nothing else to wire up.

How It Works

The extension hooks into three CDI lifecycle events:

1. Collecting injection points

<T> void collectInjectionPoints(@Observes ProcessInjectionPoint<?, T> pip) {
    // Records the type and qualifiers of every injection point
}

<T> void collectFromBean(@Observes ProcessBean<T> pb) {
    // Catches inherited generic injection points that some
    // implementations miss in ProcessInjectionPoint
}

The second observer is critical for OpenWebBeans compatibility — OWB does not always fire ProcessInjectionPoint for inherited generic fields like BaseDao<E> in an abstract superclass.

2. Registering mocks for unsatisfied types

void registerMockBeans(@Observes AfterBeanDiscovery abd, BeanManager bm) {
    for (InjectionPointKey key : collectedInjectionPoints) {
        if (isSatisfied(key, bm)) {
            continue;  // real bean exists — don't mock
        }
        abd.addBean(new MockBean<>(rawType, key.type, key.qualifiers));
    }
}

For each unsatisfied injection point, a synthetic @RequestScoped bean backed by Mockito.mock() is registered. The mock beans:

PassivationCapable

Implements the interface with a unique ID, satisfying Weld's requirement for normal-scoped beans.

Qualifier-Aware

Carries the exact qualifiers from the injection point. Follows CDI spec rules for @Default.

@Named Support

Returns the @Named value as the bean name when that qualifier is present.

Logged

Every registered mock is logged at INFO level so you can see exactly what was mocked.

Supported CDI Patterns

The extension handles the full spectrum of CDI injection patterns:

Interface, no impl @Inject Greeting Yes
Concrete class, no bean @Inject NotificationSender Yes
Parameterized generic @Inject BaseDao<Order> Yes
Constructor injection @Inject MyBean(LogService log) Yes
Initializer method @Inject void init(ValidationService vs) Yes
@Named qualifier @Inject @Named("primary") AuditService Yes, with matching name

Multiple qualifiers @Inject @Premium @Reliable ShippingService Yes, with both

@Nonbinding members @Inject @Traced(description="...") TracingService Yes, ignoring non-binding values
Producer method parameter @Produces String format(FormatService fs) Yes
Observer method parameter void on(@Observes Event e, LogService log) Yes
Satisfied by real bean @Inject @Premium HealthCheck No — real bean wins
Stereotype + qualifier Scope from stereotype, qualifier declared directly Correctly resolved
DeltaSpike partial bean @PartialBeanBinding types Skipped — DeltaSpike provides the proxy

A Complete Test

Uses only standard CDI SE APIs. No Weld helpers, no OWB utilities. 

class OrderServiceTest {

    static SeContainer container;
    static RequestContextController rc;

    @BeforeAll
    static void boot() {
        container = SeContainerInitializer.newInstance().initialize();
        rc = container.select(RequestContextController.class).get();
        rc.activate();
    }

    @AfterAll
    static void shutdown() {
        rc.deactivate();
        container.close();
    }

    @Test
    void orderServiceStartsWithMockedDao() {
        OrderService service = container.select(OrderService.class).get();
        assertNotNull(service);

        // The DAO is a Mockito mock — returns null by default
        assertNull(service.getDao().findById(1L));
    }

    @Test
    void mockedDaoSupportsStubbingWithConcreteEntity() {
        OrderService service = container.select(OrderService.class).get();
        BaseDao<Order> dao = service.getDao();

        Order expected = new Order("Widget");
        Mockito.when(dao.findById(42L)).thenReturn(expected);

        assertEquals("Widget", dao.findById(42L).getItem());
    }
}

During bootstrap, the extension logs what it mocks:

[DynamicTestBean] Registering mock for: Greeting
[DynamicTestBean] Registering mock for: NotificationSender
[DynamicTestBean] Registering mock for: BaseDao<Order>
[DynamicTestBean] Registered 3 mock bean(s) for unsatisfied injection points.

CDI Implementation Compatibility

The extension uses only standard CDI 4.1 APIs and is verified against both major implementations on every build:

ImplementationVersionProfile
OpenWebBeans4.0.3mvn test (default)
Weld5.1.4.Finalmvn test -Pweld
OWB + Java 25: OWB 4.0.3 ships with an older xbean-asm that doesn't support Java 25 class files. The OWB profile overrides xbean-asm and adds ASM 9.8 to resolve this.

Third-Party CDI Extension Compatibility

The addon is built exclusively on the standard CDI API — it has no dependency on any third-party CDI extension or specific CDI implementation. However, it detects types managed by other extensions and avoids mocking them — preventing ambiguous resolution conflicts.

Apache DeltaSpike Partial Beans

Types annotated with a @PartialBeanBinding binding annotation are automatically skipped. DeltaSpike's partial bean module provides the real proxy implementation; the addon only mocks the handler's own unsatisfied dependencies. 

@RepositoryBinding                    // carries @PartialBeanBinding — skipped by the addon
@ApplicationScoped
public interface UserRepository {
    String findUser(String id);   // implemented by RepositoryHandler
}

@RepositoryBinding
@ApplicationScoped
public class RepositoryHandler implements InvocationHandler {
    @Inject
    private NotificationService notificationService;  // ← auto-mocked
    ...
}
No DeltaSpike dependency required. Detection uses annotation name strings at runtime. If DeltaSpike is not on the classpath, the check is a no-op.

Getting Started

  1. Add the dependency (test scope): 
    <dependency>
    <groupId>org.os890.cdi.addon</groupId>
    <artifactId>dynamic-cdi-test-bean-addon</artifactId>
    <version>1.0.0-SNAPSHOT</version>
    <scope>test</scope>
</dependency>
  2. Add a CDI SE implementation to your test classpath — Weld SE (weld-se-core) or OpenWebBeans SE (openwebbeans-se).
  3. Write your test with SeContainerInitializer. The extension activates automatically via SPI. No beans.xml changes, no producer methods, no mock registrations.
