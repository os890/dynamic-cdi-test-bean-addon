@EnableTestBeans
Custom Test Implementations Without @Priority

Mockito mocks are great for most tests. But sometimes you need a hand-crafted replacement. @EnableTestBeans lets you activate CDI alternatives per test class — no @Priority, no beans.xml, no global side effects.

The Motivation

The Dynamic CDI Test Bean Addon auto-mocks unsatisfied injection points with Mockito. That works perfectly when you just need the container to start. But what if you need a specific test implementation — one that records calls, returns computed values, or validates inputs?

You could stub the Mockito mock, but that only works when you can access the mock instance directly. With CDI proxies and @RequestScoped beans, that's not always straightforward. What you really want is a proper CDI bean that implements the contract your way.

CDI has @Alternative for exactly this. The problem: alternatives need @Priority to be globally enabled, and a globally enabled alternative affects every test on the classpath. You can't activate one alternative for OrderServiceTest and a different one for PaymentServiceTest.

The Solution: @EnableTestBeans

@EnableTestBeans lets you activate @Alternative beans per test class. The extension enables them programmatically via AfterTypeDiscovery — no @Priority needed.

  1. Create your replacement as a normal CDI bean with @Alternative — define your own scope, qualifiers, and logic. Don't add @Priority. 
    @Alternative
@ApplicationScoped
public class CustomGreeting implements Greeting {

    @Override
    public String greet(String name) {
        return "Hello, " + name + "!";
    }
}
  2. Annotate your test class with @EnableTestBeans + @TestBean to activate it: 
    @EnableTestBeans
@TestBean(bean = CustomGreeting.class)
class GreetingTest {
    // Greeting injection points now use CustomGreeting
    // Everything else is still auto-mocked
}
  3. Activate multiple replacements — @TestBean is repeatable: 
    @EnableTestBeans
@TestBean(bean = CustomGreeting.class)
@TestBean(bean = CustomNotificationSender.class)
class FullIntegrationTest {
    // Both alternatives active, rest auto-mocked
}
  4. Or use beanProducer for a producer bean that provides multiple types via @Produces methods: 
    @Alternative
@ApplicationScoped
public class TestProducers {

    @Produces
    public AuditService audit() {
        return action -> "audited: " + action;
    }
}

@EnableTestBeans
@TestBean(beanProducer = TestProducers.class)
class AuditTest {
    // AuditService comes from the producer, rest auto-mocked
}

Both bean and beanProducer can be combined: 

@EnableTestBeans
@TestBean(bean = CustomGreeting.class)
@TestBean(beanProducer = TestProducers.class)
class CombinedTest {
    // Greeting from CustomGreeting, AuditService from TestProducers, rest auto-mocked
}

How It Works Internally

The extension hooks into three CDI lifecycle phases:

1. Veto unselected alternatives

During ProcessAnnotatedType, the extension vetoes any @Alternative bean that is not referenced by the active test class's @TestBean. This prevents their @Produces methods from registering beans that would conflict with auto-generated Mockito mocks.

2. Enable selected alternatives

During AfterTypeDiscovery, the extension reads @TestBean annotations from the active test class and enables each referenced alternative. The active test class is set by the internal JUnit extension (embedded in @EnableTestBeans) before the CDI container starts.

3. Mock the rest

During AfterBeanDiscovery, Mockito mocks are registered only for injection points that are still unsatisfied — the enabled alternatives already cover their types.

No @Priority, no global activation. The alternative is enabled for the current container bootstrap only. Different test classes on the same classpath can activate different alternatives — the JUnit extension scopes each @TestBean to its own test class.

Key Rules

AspectRule
Replacement bean Normal CDI bean with @Alternative (no @Priority). Use bean for direct implementations, beanProducer for classes with @Produces methods.
Scope Defined by the replacement bean itself (or @Dependent if none)
Qualifiers Defined by the replacement bean — must match the injection point
Test class annotations @EnableTestBeans + one or more @TestBean
Non-replaced types Still auto-mocked by the extension
