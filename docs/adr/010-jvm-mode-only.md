# ADR-010: JVM Mode Only for Quarkus Module

## Status

Accepted

## Context

Quarkus supports two execution modes:

- **JVM mode** — standard JVM with ArC's generated bytecode loaded at
  runtime via classloader
- **Native mode** — GraalVM native image with ahead-of-time compilation,
  closed-world assumptions, and restricted reflection

The addon uses several patterns that are incompatible with native mode:

1. **Runtime reflection** — `InlineFieldBeanCreator` reads static field
   values via `Field.get(null)`. `MockBeanCreator` receives a `Class<?>`
   parameter. The JUnit extension scans classes via `Class.forName()`.
2. **Dynamic classloading** — a custom classloader is swapped in at
   runtime to make generated `ComponentsProvider` visible.
3. **Mockito** — Mockito uses ByteBuddy for dynamic proxy generation,
   which is not native-image compatible without extensive configuration.

## Decision

The Quarkus module supports JVM mode only. Native mode is explicitly
out of scope.

## Consequences

- No GraalVM reflection configuration or substitution classes needed.
- Free use of reflection, dynamic classloading, and Mockito throughout.
- The addon is a test-time library — native mode compilation of tests
  is rare and not a practical requirement.
- If native mode support is ever needed, it would require replacing
  reflection with build-time code generation and integrating with
  Quarkus's native image configuration.
